# Coding Interview Framework

面接官と議論しながら使う6ステップのフレームワーク。

## Step 1: Clarify（入出力・制約の明確化）

問題を自分の言葉で言い換え、曖昧さを潰す。

- **Input**: 
- 入力データの条件を聞く
    - minimum size
    - maximum size
    - 型、範囲、null/空/負数は来るか？
- 入力データの条件を簡単にできるか聞く
    - 文字列の場合、英小文字のみでいいか
- **Output**: 何を返すか？形式は？複数解がある場合は？
- **Constraints**: データサイズ、時間/空間の制約、入力データの種類ソート済みか？

面接官に確認: 「この問題は〜という理解で合っていますか？」

## Step 2: Examples（具体例の列挙）

自分で例(test case)を作り、理解を実証する。

| 種類 | 目的 | 例 |
|------|------|----|
| Normal | 典型的な入力で動作確認 | [1,2,3], target=3 → [0,1] |
| Border | 境界値（0件、1件、最大） | [], [1], 上限ギリギリ |
| Edge | 特殊ケース（重複、負数、全同値） | [0,0,0], [-1,-2,3] |

## Step 3: Approach & Complexity（方針と計算量）

解法を言語化してから実装に入る。

1. **Brute Force** をまず述べる（計算量も）
2. **最適解** を提案（どのデータ構造/アルゴリズムを使うか）
3. **計算量を明示**: Time O(?), Space O(?)

面接官に確認: 「この方針で進めてよいですか？」

## Step 4: Coding（コメントファースト）

```
# 1. まずコメントで骨格を書く
# 2. 各コメントを実装に置き換える
# 3. 書きながら声に出して説明する
```

- 沈黙しない。詰まったら「少し考えさせてください」と宣言
- ヘルパー関数は「ここは別関数に切り出します」と宣言して後回しにしてよい

## Step 5: Test & Debug（検証）

- Step 2の例でドライラン（手で変数を追う）
- バグを見つけたら素直に「ここ間違ってますね」と修正
- エッジケースの動作を確認

## Step 6: Tradeoffs（トレードオフの議論）

解法の長所・短所を面接官と議論する。

- 時間 vs 空間のトレードオフ
- 別のアプローチとの比較（再帰 vs 反復、BFS vs DFS など）
- スケーラビリティ（データが巨大になったら？）
- 実務での適用（この問題の実際のユースケースは？）

---

## 困ったときの対処

| 状況 | 対応 |
|------|------|
| 解法が思いつかない | 「Brute Forceから考えていいですか？」 |
| 実装で詰まる | 「擬似コードで書いてもいいですか？」 |
| バグが見つかる | 素直に認めて直す |
| 時間が足りない | 「残り時間で〇〇だけ完成させます」 |

## 一言で覚える

> **Clarify → Examples → Approach → Code → Test → Tradeoffs**
